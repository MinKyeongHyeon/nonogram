## 장점

### 1. auth 기반 코드 즉시 재사용 가능

`store/auth-store.ts` (Zustand 세션 스토어), `components/auth-provider.tsx` (onAuthStateChange 리스너), `components/protected-route.tsx` (클라이언트 가드)가 패턴이 올바르고 바로 이식 가능했다.

### 2. 환경변수 구조가 잘 정리되어 있음

`lib/env.ts`로 환경변수를 타입-세이프하게 중앙화한 패턴이 유용했다. 빌드 타임 에러를 런타임 에러로 라우팅하는 구조가 명확했다.

### 3. 미들웨어 보안 헤더 레퍼런스로 유용

`proxy.ts`에 CSP, X-Frame-Options, Referrer-Policy 등이 미리 작성되어 있어서 OWASP 대응 헤더를 처음부터 작성하지 않아도 됐다.

### 4. Supabase + Next.js App Router 호환 패턴 확립

`createClient` 호출을 클라이언트/서버 분리하는 구조, RLS와 연계한 API 라우트 패턴이 이미 잡혀 있어 아키텍처 결정 시간이 줄었다.

## 단점

### 1. OAuth 콜백을 Server Route Handler로 처리 — 실제 버그 유발 ❌

**상황**: 보일러플레이트의 `app/auth/callback/route.ts`는 서버 Route Handler로 PKCE code를 교환하는 방식.

**문제**: Supabase 클라이언트의 `detectSessionInUrl: true` 옵션이 브라우저에서 code를 자동 소비하고 URL에서 제거한다. 서버에서도 code를 읽으려 하면 이미 소비된 상태라 `missing_code` 에러 발생.

**결론**: 콜백은 반드시 **클라이언트 컴포넌트** + `onAuthStateChange`로 처리해야 한다. 보일러플레이트의 방식은 PKCE 플로우와 충돌한다.

```ts
// ❌ 보일러플레이트 방식 (Server Route Handler)
// app/auth/callback/route.ts
const code = request.nextUrl.searchParams.get("code");
await supabase.auth.exchangeCodeForSession(code); // → missing_code 에러

// ✅ 올바른 방식 (Client Component)
// app/auth/callback/page.tsx
useEffect(() => {
  const { data } = supabase.auth.onAuthStateChange((event, session) => {
    if (event === "SIGNED_IN") router.replace(returnTo);
  });
  return () => data.subscription.unsubscribe();
}, []);
```

### 2. Rate Limit이 인메모리 — Serverless 환경에서 무의미

`proxy.ts`의 Rate Limit 구현이 `Map<string, Bucket>` 인메모리 방식이다. Vercel Serverless Functions는 요청마다 새 인스턴스가 생성될 수 있어 카운터가 초기화된다. 실질적으로 Rate Limit이 작동하지 않는다.

**필요한 것**: Upstash Redis 같은 외부 상태 저장소.

### 3. 카카오/네이버 OAuth 중심 — Google 등 범용 OAuth 처리 미흡

버튼 UI와 공급자 설정이 카카오/네이버에만 맞춰져 있다. Google, GitHub 등 다른 공급자를 추가할 때 어디를 수정해야 하는지 명확하지 않아 리팩토링이 필요했다.

### 4. 코드 재사용률이 낮음 (~30%)

포함된 파일 중 실제로 이식한 것:

- `lib/supabase-client.ts` ✅
- `store/auth-store.ts` ✅
- `components/auth-provider.tsx` ✅
- `components/protected-route.tsx` ✅
- `proxy.ts` (구조 참고만) ⚠️

사용하지 않은 것:

- `toss-payment-widget.tsx` ❌
- `lib/toss-server.ts` ❌
- `components/policy-consent.tsx` ❌ (로직 제거 필요)
- Naver OAuth 관련 코드 전체 ❌
- DB 마이그레이션 스크립트 ❌

### 5. `protected-route.tsx`에 정책 동의 로직 포함

비로그인 시 리다이렉트 외에 "약관 동의 여부" 체크 로직이 섞여 있었다. 단순 인증 가드로 사용하려면 이 로직을 제거하고 단순화해야 했다.

### 6. `supabaseClient` 빌드 타임 초기화 문제 미해결

보일러플레이트는 모듈 레벨에서 `createClient()`를 직접 호출한다. Next.js 빌드 타임에 환경변수가 없어서 `Invalid supabaseUrl` 에러가 발생했다. Proxy 패턴 또는 lazy init으로 교체가 필요했다.

## 보완하면 좋을 것들

### 핵심 수정

1. **OAuth 콜백을 Client Component 방식으로 교체**
   - `app/auth/callback/route.ts` → `app/auth/callback/page.tsx`
   - `onAuthStateChange`로 `SIGNED_IN` 이벤트 대기 + 타임아웃 처리

2. **supabaseClient lazy init**

   ```ts
   // 모듈 레벨 createClient() 제거
   // Proxy 패턴 또는 getSupabase() 팩토리로 교체
   let _client: SupabaseClient | null = null;
   export function getSupabase() {
     if (!_client) _client = createClient(env.supabaseUrl, env.supabaseAnonKey, { ... });
     return _client;
   }
   ```

3. **Rate Limit — Upstash Redis 연동**
   ```ts
   import { Ratelimit } from "@upstash/ratelimit";
   import { Redis } from "@upstash/redis";
   ```

### 구조 개선

4. **OAuth 공급자를 config 배열로 관리**

   ```ts
   const OAUTH_PROVIDERS = [
     { id: "google", label: "Google", icon: "..." },
     { id: "kakao", label: "카카오", icon: "..." },
   ];
   ```

   공급자 추가 시 config만 수정하면 되도록.

5. **Toss Payments를 optional 모듈로 분리**
   결제 기능이 없는 프로젝트에서는 관련 파일이 불필요하다. `payments/` 디렉토리로 격리하고 선택적으로 포함하도록.

6. **`protected-route.tsx` 단순화**
   정책 동의 로직 제거, 순수 인증 가드만 남기기:

   ```tsx
   // 로그인 여부만 체크하는 단순 버전
   if (!session) router.replace(`/login?returnTo=${pathname}`);
   ```

7. **README에 "이 파일만 쓰세요" 가이드 추가**
   보일러플레이트 전체를 복붙하는 게 아니라 필요한 파일만 cherry-pick할 수 있도록 사용 가이드 명시.

## 총평

좋은 레퍼런스 코드이지만 **"그대로 쓰면 버그 나는" 코드**가 포함되어 있다. 특히 OAuth 콜백 방식은 실제 프로젝트에서 수 시간의 디버깅을 유발했다. 핵심 auth 패턴(세션 스토어, AuthProvider)은 재사용 가치가 높으니, 문제가 있는 콜백 처리와 Rate Limit 구현만 수정하면 훨씬 신뢰할 수 있는 보일러플레이트가 될 것이다.
